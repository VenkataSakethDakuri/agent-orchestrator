Design Document: Hybrid Cloud Agents for Agent Orchestrator (AO)

Draft · rev.3 — hosted coordinator, separate database, auth + runner protocol

Move the coordinator off the laptop, make the database a separate always-available component, and run agents locally or in cloud sandboxes — so cloud runs persist 24/7 no matter which device is open.

1. Context and Scope

Background / Problem Statement

AO runs coding agents in parallel git worktrees and shepherds their PRs. Today it is local-first in two ways at once, and both must change:

  • Execution is local — agents run as processes on the daemon's machine (tmux/conpty + local PTY). Your laptop is the ceiling.
  • The coordinator IS the laptop — the daemon runs on your machine and its state is an embedded SQLite file on that disk (~/.ao/data/ao.db, storage/sqlite/db.go:56). It's a loopback, no-auth service (httpd/cors.go:13).
The decisive failure. You launch a cloud agent, then close your laptop. The agent keeps working on the remote VM — but the daemon (the only DB writer) and its database are on the sleeping laptop. Nothing can be persisted, and no other device can watch it. A long-lived, remotely-executing system cannot depend on a client that sleeps.

This revision moves the coordinator off the laptop and makes the database a separate, always-available component, so cloud runs persist regardless of any one device. The laptop becomes a client you open when you want to look — the same principle every durable system follows: state lives with a long-lived, always-on coordinator, not with an ephemeral client.

Goals

G1 — Always-on coordinator. Host the daemon on an always-on host (not the laptop). Success = a cloud agent keeps running and persisting while every operator device is offline; reopening any client shows current state.
G2 — Database as a separate component. Move durable state from an embedded on-daemon file to a standalone networked PostgreSQL, so DB availability/scaling/backup are independent of the daemon.
G3 — Hybrid, pluggable execution. Run any session local or cloud behind a SandboxProvider port (SSH-to-VM deep; Daytona/E2B/Modal/Docker/k8s same-interface), SCM/PR loop unchanged.

Non-Goals

  • Not a public multi-tenant SaaS. V1 is self-hosted, single-operator AO — you run the (hosted) daemon on infrastructure you control, for yourself/your team; it is not an AO-hosted service for many mutually-untrusting customers. SaaS would require tenant isolation, billing/metering, org-auth (SSO/RBAC), per-tenant quotas, abuse controls, and much stronger secret isolation — a distinct system, not a config flag. Single-operator is still multi-device (laptop + phone + CLI reach the one hosted daemon); multi-operator/SaaS is out of scope but no longer precluded.
  • Not running the LLM ourselves — agents call their own model providers.
  • Not implementing every provider in V1 — only ssh.
  • Not keeping loopback/no-auth — exposing the daemon requires auth (§3); this design explicitly revises that invariant. (Auth = "prove it's the operator across devices," not the org-auth/RBAC a SaaS needs.)

2. Architecture and High-Level Design

The three-tier shape

The revision splits AO into three independently-hosted tiers — the standard control-plane / data-plane / client separation:

Coordinator (daemon)

Always-on host. Owns business logic, the SCM/PR loop, CDC fan-out, runner-ingress, and the client API. Holds no durable state itself; runs no agent code.

Database (separate)

A managed PostgreSQL on its own host — the single source of truth, reached by the daemon over the network. Own uptime, backups, scaling.

Clients (many, thin)

Electron / CLI on laptop, phone, etc. Pure viewers/drivers over authenticated HTTPS + SSE. Closing them changes nothing. Execution is orthogonal: each session runs local (on the coordinator host) or cloud (in a sandbox that dials back). The DB stays authoritative regardless of where the agent runs or which client is open.

The execution seam (unchanged)

AO already funnels all execution through runtimeselect.Runtime (ports.Runtime + Attacher + SendMessage + GetOutput) and ports.Workspace. A RemoteRuntime + RemoteWorkspace satisfy those exact interfaces, so Manager.Spawn (manager.go:191) is untouched. A small aorunner on the sandbox reuses AO's own tmux/gitworktree adapters locally and dials back a reverse WS tunnel (coder/websocket) to the hosted daemon — which, being always-on, always has a home for it.

Runner protocol (aorunner contract)

One coder/websocket connection per session (runner→coordinator) carries two multiplexed channels: RPC (control) and PTY (live terminal).

Commands (RPC)

The coordinator sends the same Runtime/Workspace calls it uses locally (create/destroy the workspace and process, probe liveness, send input) down the tunnel; the runner sends back register/status/exit events. Just request/response over the socket — the port methods, now remote.

PTY streaming

The agent runs in tmux on the VM; aorunner relays its terminal as a live byte stream over the tunnel into AO's unchanged terminal.Manager → xterm.js, with keystrokes/resize flowing back. tmux gives persistence — a dropped tunnel doesn't kill the terminal; aorunner re-attaches on reconnect.

Heartbeat (CO ↔ VM — "Link B")

ping/pong 30 s / 3-miss (~90 s) = dead. CO misses runner → VM dead → reap + StashUncommitted. Runner misses CO → buffer mode. (Client↔CO keepalive only affects UI freshness.)

Reconnect & buffering

Newest-wins: re-dial + idempotent hello supersedes a stale connection. The offline buffer lives at the edge — on the VM, not the DB (the DB may be down); durable events + PTY output carry seq numbers and replay from the last CO-acked seq (like SSE Last-Event-ID).

Backpressure — drop oldest (split)

Drop oldest for the live PTY stream + logs; never drop durable events (session state, results, PR facts) — flow-controlled + replayed.

Limits & auth failure

Ring-buffer scrollback (~10k lines) + logs (~1–2 MB/session) with a visible truncation marker. Auth: hard failure (revoked/expired token) → close, no retry; transient network error → retry w/ backoff.

Data Model & keying

Everything stays keyed by conversation (session) id — sessions.id PRIMARY KEY, with children (pr, pr_checks, change_log, new sandboxes) referencing session_id ON DELETE CASCADE. sandboxes is PK session_id (1:1 to the conversation). This is what makes hybrid safe: execution moves, state does not — the record lives in the separate DB regardless of where the agent ran or which client is open.

Storage: separate PostgreSQL (replacing embedded SQLite)

Decision: promote the database to a standalone networked PostgreSQL component — required by G1/G2. You cannot have an always-on coordinator persisting cloud-run state if the store is an embedded file on a client machine.

ConcernBefore (SQLite)After (Postgres, separate)
Location~/.ao/data/ao.db on the laptop diskOwn managed instance, over the tailnet

Migration cost (honest). AO's sqlc.yaml targets engine: "sqlite" and CDC relies on SQLite triggers feeding an in-process poller (cdc/broadcast.go). Postgres means: (1) a Postgres sqlc target + regenerated gen/; (2) porting goose migrations to Postgres dialect; (3) re-implementing the change-feed with LISTEN/NOTIFY (or logical replication); (4) a network DSN + pool + retry/backoff. This is the largest single work item — hence the headline of this revision.

3. Operational Requirements

Security & Privacy

  • Deployment shape. The coordinator (AO daemon, containerized) runs on an always-on host; the database is a separate PostgreSQL instance with its own lifecycle/backups; aorunner is a static Go binary needing only git+tmux on the sandbox; clients (Electron/CLI) point at the hosted daemon.
  • Recommended networking — Tailscale. Join the coordinator, operator devices, and sandboxes to a Tailscale tailnet (or Headscale/WireGuard for a pure-OSS self-hosted control plane). Sandboxes join via an ephemeral auth key injected at SSH bootstrap (auto-removed on teardown). WireGuard already encrypts node-to-node, so a separate TLS layer is not required; optionally use Tailscale MagicDNS HTTPS if the web UI needs a browser secure context. Nothing is public.
  • AO auth still mandatory (httpd/cors.go:13 loopback/no-auth ends). Tailscale authenticates devices onto the network, not the operator to the app. Since each user runs their own orchestrator, AO auth is how the operator logs in and authorizes their GitHub token and how multi-device works — a compromised tailnet node still can't drive the orchestrator. See Authentication below.
  • DB security: Postgres over the tailnet, least-privilege creds via secret manager, not exposed to the public internet — only the daemon reaches it.
  • Bootstrap secrets: model/git tokens injected into the sandbox at SSH time; never persisted; sandbox destroyed on teardown.
  • Isolation win: agents run on disposable remote boxes, not any operator device.

Authentication

Where auth is required — seven distinct boundaries. Tailscale (#7) is the network perimeter; app-layer auth is identity + credential binding on top.

Boundaries 2–7 are settled elsewhere; the decisions below are for #1, Client → Coordinator.

Identity — built-in default, OIDC optional

  • Built-in accounts (email + argon2 hash) — the default: zero external dependency, right for a solo self-hosted operator.
  • OIDC — optional pluggable provider (OSS: Authelia / Keycloak / Dex, or Google/GitHub/Okta) for teams wanting SSO/MFA. The multi-operator path; out of scope for V1, not precluded.

ao login — local browser, no code

The CLI runs on the operator's own machine, so login is a fully-automatic loopback OAuth flow — nothing is typed or pasted:

  1. ao login starts a temp server on a random localhost port + opens the browser (PKCE + state).
  2. Operator authenticates in the browser.
  3. Coordinator redirects to localhost:<port>/callback?code=…; the CLI catches it and exchanges for tokens automatically.
  4. "✓ logged in." PKCE + state secure the exchange.

Token storage

Two tokens: short-lived access (every request) + longer-lived refresh (mints new access tokens).

  • Electron: refresh token in the OS keychain via safeStorage; access token in memory only.
  • CLI: token in the OS keychain, 0600 file fallback.
  • Never plaintext / localStorage.

Session expiry & revocation

  • Expiry: access ~1 h (silently auto-refreshed); refresh ~30 d sliding (idle 30 d → fresh ao login).
  • Revocation: DB-backed, per-device (fits AO's durable-facts + CDC model) → delete/flag the row, next request fails instantly. Revoke a lost laptop without dropping the phone; web UI lists active devices. Runner tokens auto-revoked at teardown.
Monitoring & offline behavior. Surface sandboxes.status via CDC. DB health: alert on pool exhaustion, replication lag, disk/backups — the DB is now a first-class dependency with its own SLOs. Coordinator uptime: alert on daemon downtime. Offline: with the coordinator hosted, operator devices going offline is a no-op for persistence; if the coordinator is briefly down, runners buffer and replay on reconnect, and the agent's git push to GitHub is independent of AO regardless. Orphan sweep terminates sandboxes rows with no live session.

4. Alternatives Considered

Alt 1 — Keep daemon + embedded SQLite on the laptop (rev.1).

Rejected: close the laptop → cloud runs can't persist and nothing is reachable from other devices. Exactly what this revision fixes.

Alt 2 — Host the daemon but keep embedded SQLite co-located on it.

Rejected as default (kept as fallback): solves availability but ties DB backups/scaling to the daemon host and blocks multi-coordinator. Fine for a solo user; Postgres-as-separate satisfies G2 too.

Alt 3 — Daemon dials OUT, no auth.

Rejected: fails on NAT'd/ephemeral sandboxes and awkward PTY; once the daemon is hosted, auth is required anyway. Dial-back + auth is uniform. (SSH-dial-out kept as degraded fallback.)
